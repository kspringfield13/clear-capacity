# NOTES.md — Architectural learnings for the ClearCapacity autonomous loop

Cross-run memory for the autonomous agents (hourly improver, 6-hour UX curator) and human readers.
Cloud sessions are stateless — without this file, every run relearns the codebase from scratch.

**Read this file right after STATUS.md. Append durable learnings here before you commit.**

## How to maintain this file
- Record only DURABLE, reusable insights: established patterns, conventions, gotchas, non-obvious decisions — things that save the next run time.
- Do NOT duplicate CLAUDE.md (build commands, repo layout) or STATUS.md (the task backlog). Refer to them instead.
- Keep it tight and curated (aim < ~150 lines). Update or delete entries that have gone stale; don't append forever.

## Established UI patterns — reuse these, don't reinvent
- **Async op state** → use the `useAsyncStatus` hook (`apps/desktop/src/hooks/useAsyncStatus.ts`): `start()/fail()/reset()` instead of paired status+error `useState`. All five AI operations already use it.
- **Empty states** → route every screen's empty case through the shared `EmptyState` component (icon + headline + primary CTA). Don't write ad-hoc empty markup; its `children` slot renders into `.empty-state-actions` (already full-widths/stacks at ≤600px) — put navigation buttons (`primary-action`/`secondary-action`) there. The empty `daily`/`weekly`/`forecast` screens all use "Import calendar"/"Open Settings" buttons → `onOpenScreen("setup")`.
- **Onboarding checklist** → `buildOnboardingSteps(status)` + `<OnboardingCard>` (`components/common/OnboardingCard.tsx`) are the single source for the getting-started steps (track → import → configure AI → classify). Both `SetupScreen` (inline, `!allDone`) and the first-run card on empty daily/weekly render it; the card takes an optional `onDismiss` (renders the `X`). First-run gating lives in `App.tsx` as `showOnboarding = !isDemoMode && !onboardingDismissed && blocks.length === 0`, threaded through `ScreenRouter`. Don't re-inline the step list anywhere — extend the helper.
- **Inline error recovery** → use the shared `components/common/InlineError.tsx` (`<InlineError message={err} onRetry={handler} />`) — a single `role="alert"` `.error-row` with an `AlertTriangle` icon + `.inline-error-text` box + optional "Try again" button. This is the ONE source for persistent in-panel AI-op errors (ReviewCopilot/Forecast/Narrative/classification all route through it); don't hand-roll an `.error-row`/`.error-retry` block. Style lives in `.inline-error-text` (light/dark) — the old per-panel `.copilot-error`/`.forecast-error`/`.narrative-error` classes were removed. Gotcha: `.narrative-screen` uses `gap: 0`, so it needs a scoped `.narrative-screen .error-row { margin-bottom: 14px }` rule to keep spacing (other panels get spacing from their parent's grid `gap`). Passive informational errors with no idempotent retry (capture/visual-context) stay as bare `.capture-error` paragraphs — the boxed InlineError style is reserved for *actionable* (retry-able) errors. Complementary to toasts (transient), not a substitute — don't double-render both for one event.
- **Loading** → use the `skeleton-shimmer` animation + `.skeleton-line` / `.skeleton-block` utility classes so the layout doesn't shift when results arrive (don't show bare "Generating…" strings).
- **Per-screen toolbar action** → the primary action per screen is wired via the screen-aware `toolbarActions` logic in `App.tsx`.
- **Confirm/alert dialogs** → use the shared `components/common/ConfirmDialog.tsx` for any destructive/irreversible confirm — don't hand-roll another modal. It is a `role="alertdialog"` + `aria-modal` panel with the loop's a11y baseline already built in: `useId`-keyed `aria-labelledby`/`aria-describedby`, a Tab/Shift+Tab focus trap, Esc + overlay-click (`onMouseDown` target===currentTarget) cancel, initial focus on the safe **Cancel** control (not the destructive confirm), and focus restore to the trigger via a mount-time `document.activeElement` capture restored in the effect cleanup. Props: `title`/`description`/`confirmLabel`/`cancelLabel`/`tone` (`"danger"` → `.dialog-confirm-danger` white-on-`--danger`)/`onConfirm`/`onCancel` + a `children` slot. Render it conditionally from a local `useState` boolean; the trigger restores focus automatically when it unmounts. Styling is `.dialog-*` (overlay/panel/title/desc/delete-list/export-action/actions) — token-based, 420px-safe (`flex-wrap` actions), fade-in auto-zeroed by the global `prefers-reduced-motion` reset (no per-component guard needed).
- **Screen navigation** → ⌘1–⌘9 keyboard shortcuts (`SCREEN_KEYS` in `App.tsx`: 1 daily … 9 setup) live in `App.tsx`; respect the input/textarea focus guard when adding any new key handler.
- **Transient feedback (toasts)** → use the `useToasts` hook (`hooks/useToasts.ts`) + `<ToastHost>` (`components/common/ToastHost.tsx`) for app-level success/error/info that should auto-expire — NOT for persistent in-panel errors (those use the inline `.error-row`/`error-retry` pattern; the two are complementary, don't double-render the same Retry). The queue is owned in `App.tsx` (single source of truth); `pushToast` (stable `useCallback`, safe in effect deps) threads down through `ScreenRouter` props, and the visual stack renders once in `AppShell` (`toasts`/`onDismissToast`). To surface a swallowed AI error as a toast, add an effect keyed on the error string with a `prev*Error` ref guard (`error && error !== prev.current` → push; always write `prev.current = error`) so an unchanged value never re-announces but a genuine new failure fires once — `useAsyncStatus.start()` nulls the error before `fail()`, which makes the ref toggle work. `ToastInput`/`PushToast`/`Toast` types are exported from the hook. Stack caps at 4 (clear the evicted timer when slicing). Slide-in is auto-zeroed by the global reduced-motion reset — no per-component guard.

## Conventions
- **Styling**: Vercel Geist design tokens via CSS variables in `apps/desktop/src/styles.css`. Support light AND dark. NO hardcoded hex — add a token with light/dark variants instead.
- **AI-provider form help is preset-driven**: per-provider inline help in the Advanced Settings form (`SetupScreen.tsx`) lives ENTIRELY on the `AIProviderPreset` (`services/aiProviders.ts`), never hardcoded in the component — `modelNote`/`baseUrlNote`/`visionNote` (helper `<small>` under each field, styled by the shared `.ai-field > small` rule), `modelSuggestions?: string[]` (click-to-fill "RECOMMENDED" chips under Model; render falls back to `[model]`, empty for custom; chip `aria-pressed={draftConfig.model === id}` is the non-color selection cue), and `docsUrl?: string` (external "{label} model docs" link, `--info` blue, `ExternalLink` + `target=_blank rel="noopener noreferrer"`). To add help for a new provider/field, extend the preset object — don't add JSX. Header gotcha: the `.ai-provider-header` is `justify-content: space-between`, so the docs link + the "stored locally" note must be wrapped together in `.ai-provider-meta` or `space-between` strands the note in the center.
- **AI prompts**: the five prompt builders in `apps/desktop/src/services/*Prompt.ts` are version-stamped; bump the version string when you change a prompt's behavior.
- **Domain types**: shared types (`WorkBlock`, `ActivitySession`, `AIConfig`, `AuditEvent`, …) live in `packages/domain/src/models.ts`; capacity/grouping logic in `packages/inference/src/`.

## Gotchas
- **Render for visual checks**: the React UI runs in a plain browser via `npm run dev` (Vite, 127.0.0.1:5173); append `?demo=1&screen=<daily|weekly|narrative|ledger|audit|setup>` for synthetic data. Tauri APIs are absent in the browser — demo mode is how you render screens headless.
- **Headless dark mode**: the app does NOT honor `prefers-color-scheme`. Theme is `document.documentElement.dataset.theme`. Setting `localStorage["clear-capacity:theme"]` via `addInitScript` is NOT enough — `App.tsx` inits `theme` state to `"light"` and its mount effect overwrites both `dataset.theme` and the stored key back to light (see theme-persistence bug below). The reliable Playwright recipe: `goto` + wait, then `page.evaluate(() => { document.documentElement.dataset.theme = 'dark'; })` AFTER the app has mounted, then screenshot. (`colorScheme: 'dark'` alone does nothing.)
- **Theme flash on first paint (cosmetic)**: `main.tsx` sets `dataset.theme = "light"` synchronously then async-patches it; `App.tsx`'s mount effect briefly resets to "light" before hydration resolves. Cosmetic only — preference is now correctly persisted (see fix below).
- **Playwright in the sandbox**: the chromium CDN download is blocked, but a usable browser ships at `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` — pass it as `executablePath`. Install the `playwright` npm module under /tmp, never in the repo.
- **Icon-only button a11y convention**: chrome/icon buttons should carry both `title` and `aria-label` (plus `aria-pressed` for toggles). The theme toggle in `AppToolbar.tsx` is the reference; match it when adding icon controls.
- **Screen-reader live regions**: use `role="alert"` on a conditionally-rendered element (or a persistently-rendered element whose content changes) to announce validation errors to screen readers. The `.sr-only` utility class in `styles.css` (1×1px, clipped, absolute, `border: 0`) hides the element visually without removing it from the accessibility tree — use it for any visually-hidden but SR-meaningful text.
- **Verification gate**: `npm run build` (`tsc -b && vite build`) must pass before committing. There is no automated test suite.
- **Out of loop scope**: `apps/desktop/src-tauri/` (Rust), `.env`, and the 5173 port config are off-limits to the frontend loop. Anything needing Rust (model params, request bodies, JSON schemas, per-command model routing) is a manual follow-up — leave a clear note in STATUS.md.
- **Branch protection on main**: direct `git push origin main` returns HTTP 403. All loop commits must go via an `improve/<slug>` branch + PR (GitHub MCP `create_pull_request`). The STATUS.md "Never" section already requires this pattern.
- **npm install required in fresh containers**: `node_modules` is absent in new cloud sessions; run `npm install` before `npm run build` or the tsc step fails with "Cannot find module 'react'" errors on every file.
- **Stale local origin/main**: cloud sessions clone with a local `origin/main` that may lag behind GitHub. Always `git fetch origin main` at the start of a run before branching — otherwise `improve/*` branches will be based on an old commit and `git show origin/main:FILE` will differ from the working tree. If STATUS.md in the working tree doesn't match `git show HEAD:STATUS.md`, the repo was cloned at a stale commit; fetch + reset before proceeding.
- **STATUS.md corruption risk**: curator-pass PRs that update STATUS.md can silently revert it if the curator's branch was based on an older main. If STATUS.md on main looks stale (old Done items, no recent improvements), check `git log --oneline origin/main` against CHANGELOG.md — the CHANGELOG is the reliable source of truth for what's shipped. Reconstruct STATUS.md from it.
- **Verify the task is actually unimplemented before coding**: curator passes (and STATUS duplicates) routinely re-add items already fixed in code — a single run found the top 3 UI items already solved (agent-starters `flex-start`, capture panel already a closed `<details>`, current-block already picked by latest `end_time` via `reduce`). Always open the referenced file/lines FIRST and confirm the bug still exists; if already fixed, move it straight to Done with a "verified already solved" note and pick the next genuinely-open task instead of writing a redundant diff.

- **Don't audit-log high-frequency background effects** (SOLVED for retention auto-expiry): the audit trail is capped at `.slice(-1000)`, and `activeWindowSamples` accrue one-per-native-event, so an effect that logs on every sample change (e.g. logging each retention expiry as samples continuously age past a cutoff) floods the trail and evicts genuinely meaningful events (classifications, corrections). Rule: log the DISCRETE user action that sets the policy (e.g. `changeRetentionDays` → one `retention_policy` event per window change), NOT the recurring background consequence. Auto-expiry/cleanup effects should mutate state silently. Pattern for the effect itself: filter + `return kept.length === current.length ? current : kept` inside the functional update so it returns the SAME reference when nothing crosses the boundary (no re-render loop even though the effect deps include the array it mutates), and early-return in demo mode.

## Recurring UX rough edges (watch for these)
- **Week section has FOUR sub-tabs now (Capacity / Forecast / Trends / Summary)** — the `Trends` screen (`components/capacity/TrendsScreen.tsx`) is a multi-week line chart over `snapshotHistory` (allocated/reactive/deep-work/reliable, pure SVG, `role="img"` + per-dot `<title>`, labeled `.trend-legend`) plus the forecast track record below it. **Panel-home convention (SOLVED — don't re-flag):** `ForecastTrackRecord` now lives ONLY on `TrendsScreen` (the per-week predicted-vs-actual history pairs with the multi-week chart); it was removed from `ForecastScreen`, which keeps the `ForecastAgentPanel` accuracy banner + trend line (those are about *next* week). `forecastTrackRecord` is still a `ScreenRouterProps` field but is passed to the Trends call site only — when adding a history-flavored panel, the Trends tab is its home; don't also render it on Forecast. Also: the Trends subtitle and `ForecastAgentPanel`'s accuracy line both surface the same `ForecastAccuracyTrend` — keep their copy consistent ("over the last N weeks", with a singular guard).
- **Don't render raw enum/snake_case values in the UI**: domain enums like `PrivacyLevel` (`local_only`/`derived_only`/`excluded`) and `PlannedStatus` (`planned`/`unplanned`/…) must pass through a humanizing helper in `lib/format.ts` (`auditTypeLabel`/`plannedStatusLabel`/`privacyLevelLabel`) before display — never `{event.privacy_level}` directly. Note `WorkCategory`/`WorkMode` are stored as already-human strings in `taxonomy.ts`. `CorrectionsScreen` and the `user_correction` audit-event summary in `App.tsx` both now humanize via `humanizeCorrectionValue(correction.field, …)` (fixed). Lesson: when a value is humanized for display, also humanize any pre-baked `summary`/`title` string stored on the audit event — humanize at event-construction time, not just at render. **This includes the demo seed**: `services/demoData.ts` hardcodes audit summaries directly (e.g. the `user_correction` event used a raw `"planned -> unplanned"`), so a seed string bypasses the live `addCorrection` humanizer and renders raw in demo mode while the Corrections screen shows it humanized. Demo mode is the showcase a new user/evaluator sees first — keep seed summaries consistent with the humanized live path (`lib/format.ts` helpers).
- **Group paired nav controls**: when chevrons/prev-next controls straddle a long headline (e.g. `.week-nav` in `WeeklyCapacityScreen`), they drift apart and the disabled one becomes invisible. Keep paired controls adjacent and give `:disabled` a visible low-opacity + `cursor: not-allowed` state.
- **Intensity-coded grids need a legend AND non-visual semantics**: `data-level` heatmap cells (`ActivityHeatmap`) need a "Less → More" key (done) plus per-cell `role="img"`+`aria-label` and a grid summary — hover `title` alone is invisible to keyboard/SR. They also look broken when sparse: the full 7×24 grid renders whenever there is ≥1 session, so a single active day reads as an empty box — add a low-data caption.
- **Numeric metric rows need scale + consistent units**: bare numbers like the `RiskRow` magnitudes ("33/90/10/46") read as ambiguous without a "/100" hint or a `title` tooltip. The raw-count-vs-index mismatch is now SOLVED in `RiskRow` and is the reference pattern: "Active blockers" passes `displayValue` (raw count, no `/100` hint), `dangerActive` (red bar + `.risk-blocker-count` number class), and a count-specific `aria-valuetext` — do NOT re-add this as a bug. Reuse the same trio for any future count-among-indices row. Severity-by-value tinting is also SOLVED: `RiskRow.tsx` now sets `data-severity="low|mid|high"` on the fill span (thresholds <34/34–66/≥67); `styles.css` has matching light + dark rules (low: muted blue-gray, mid: amber, high: orange); `dangerActive` row keeps its inline red override which beats any CSS rule. Apply the same `data-severity` pattern to any future value-coded bar. Same rule for chips and bare metric figures: a labeled percentage on a card needs a hover tooltip (`title`) stating *what it measures*, plus an `.sr-only` mirror so it isn't hover-only. `ConfidenceChip` is the SOLVED reference — it carries `title="…% classification confidence"` and renders 0 as "Unscored" (not "Needs review 0%"); don't re-add it. The Weekly `.summary-score` "Summary confidence" card (`WeeklyCapacityScreen.tsx`) and the BlockCard `.block-capacity` "% of week" figure now both carry `title` + `.sr-only` — SOLVED, do not re-add.
- **Color-only legend keys need hover crosslinks**: prefer bidirectional hover-highlight between legend rows and bar segments (dim non-hovered peers to ~0.3–0.35 opacity with a 0.12s transition) over relying on color alone. The SOLVED reference is the Weekly `capacity-model` legend + `StackedBar` (`WeeklyCapacityScreen.tsx` `hoveredCategory` state → `onHoverCategory`/`opacity` on both legend rows and segments). `categoryColors` palette now has no near-duplicate hues — keep it that way when adding categories. **Gap (tracked in STATUS):** the `TrendsScreen.tsx` 4-line SVG chart is the remaining color-coded multi-series viz WITHOUT this crosslink (its lines cross, so isolating one series is hard) — extend the same `hoveredSeries`-dims-peers pattern to it.
- **Decorative indicators must NOT be `<button>`s, and a metric label must map to ONE number app-wide** (2026-06-29 visual pass): two recurring clarity traps. (1) A non-interactive status glyph rendered as `<button type="button">` with no `onClick` (e.g. the Agent composer `.agent-attach` paperclip — `title` "Context is attached automatically") still gets pointer cursor + focus ring + tab stop, so users click it expecting an action and nothing happens. Use a `<span>` + `aria-label` for decorative/status glyphs; reserve `<button>` for controls that DO something. **SOLVED** for `.agent-attach`: it is now a `<span role="img">` (out of the tab order, no hover/focus ring; the `:hover` rule was deleted) keeping its `aria-label` as the accessible name. (2) **SOLVED** — the Forecast Agent hero card label collision is fixed: `ForecastAgentPanel.tsx`'s first `.forecast-summary` card now reads "AI reliable estimate" (was "Reliable new-work capacity"), distinct from the *deterministic* "Reliable new work" metric on Weekly/sidebar/forecast-header. Rule still applies generally: keep deterministic-model labels and AI-scenario labels textually distinct (the `.forecast-baseline-note` only reconciles the numbers, not the label) — don't re-introduce a shared label across the deterministic vs AI surfaces.
- **Narrow-layout header actions must opt into the full-width media query**: the `@media (max-width: 600px)` block in `styles.css` full-widths `.primary-action`/`.secondary-action`/`.header-actions`/`.review-header-actions`/`.summary-score`/`.search-box` (verified clean at 420px on Daily + Narrative). Any NEW header-action wrapper class is an orphan until added — its buttons each take `width: 100%` in a non-wrapping flex row and the trailing one overflows off-screen. When adding a per-screen header action group, add its class to that rule and stack it (`flex-direction: column`).
- **Native `<select>` truncates its current value**: in equal-width grids like `.tag-grid` (`BlockCard.tsx`) a long option label is clipped at rest. The convention is to set `title={...}` on every select reflecting the *current* value (humanized for enum fields via `lib/format.ts`) so hover reveals it — but `title` is invisible to keyboard/touch, so for the longest field prefer giving its column more width over relying on the tooltip alone. Separately, `title` reflects the *current value* — it is NOT a field-name label, so a relabel select in a header-less row (the three in `BlockCard.tsx`) has no accessible name telling SR users which field they're editing; give each such select a static `aria-label` naming the field ("Work category" / "Planned status" / "Work mode") in addition to the value `title`.
- **Stacked-bar segments need a remainder + per-segment a11y label**: `StackedBar` only renders the colored category segments; when they sum to <100% the container background shows as an unexplained gray tail. Render an explicit "Unallocated / buffer" remainder span and give every segment an `aria-label` (hover `title` alone is invisible to keyboard/SR users). Same applies to `RiskRow` magnitude bars — use `role="meter"` + `aria-valuetext`.
- **Audit-type pills are keyed by a per-type CSS class**: `AuditEventRow` sets `className={`audit-badge ${event.type}`}`, so EVERY `AuditEventType` needs a matching `.audit-badge.<type>` rule in `styles.css` with a *tinted* background + colored text (light AND dark). The convention is one distinct hue per type (sky/violet/teal/amber/blue/green/red/pink…); a colorless `var(--text)`-on-transparent rule reads as un-styled and breaks the set. **Adding a new `AuditEventType` is a 4-file recipe** (worked example: `onboarding`): (1) the union in `packages/domain/src/models.ts`; (2) the *exhaustive* `Record<AuditEventType,string>` in `auditTypeLabel` (`lib/format.ts`) — TS fails the build if you miss it; (3) the `AuditFilter` union + `filters` chip array + `filterMatches` record in `AuditLogScreen.tsx` (optional but every existing type has a filter); (4) the tinted badge rule. Styles gotcha: there are THREE `.audit-badge.<type>` blocks in `styles.css` — a base set (~3868, light-on-dark colors) that is OVERRIDDEN at equal specificity by a later light block (~5751) and a `html[data-theme="dark"]` block (~6126). The operative pair is the light (~5751) + dark (~6126) blocks; add your rule to BOTH of those (the base block is effectively dead). Existing badges use hardcoded hex here, so match the sibling hex style rather than tokens for consistency.
- **Internal-scroll lists + single-line rows break at narrow (≤600px) — audit every screen at 420px viewport**: the app shell is a fixed-viewport `.screen` (body/`.app`/`main` are height-capped, `overflow: hidden`; a real phone sees ~900px, NOT the tall fullPage screenshot — capture viewport-only shots too). Two failure modes recur on compact: (1) **List starvation** — a screen that stacks a tall header (heading + search + cards + collapsible panels) above an `overflow-y: auto`, `flex: 1 1 auto` list gives the list ~0px once the header fills the viewport, so the primary content is an unusable near-zero scroll box. Fix by relaxing the fixed-height/internal-scroll model so the page scrolls naturally (Agent/Forecast already scroll naturally — match them). **SOLVED for `.ledger-screen`**: the `@media (max-width: 760px)` block already sets `.ledger-screen { display: block }` + `.screen { overflow-y: auto }` (since PR #34), so at ≤600px `.ledger-list` flows to full content height (`flex` inert under a block parent) and the screen scrolls — verified at 420/580px (`list.clientHeight===scrollHeight`). Don't re-flag the ledger; apply this `display:block` reflow to any *new* fixed-shell screen instead. (2) **Single-line row clipping** — a `display: flex; white-space: nowrap` row (e.g. `.corrections-list li` / `.history-list li`) stays its full desktop width inside the narrow scroll container and clips its right-most content (the `old → new` change) off-screen; measure with `li.scrollWidth` vs `list.clientWidth`. Fix by making the row `flex-wrap: wrap` or stack (`flex-direction: column`) in the narrow media query. Both are `styles.css`-only. **Nested gotcha (SOLVED for `.corrections-list`)**: stacking the OUTER `li` into a column is NOT enough when an inner segment is itself a non-wrapping flex row of long values — `.correction-change` is an `inline-flex` of two `white-space: nowrap` `.correction-old`/`.correction-new` spans, so even after the li stacks, the change line measures ~406px inside a ~388px list at 420px and clips. The inner row ALSO needs `flex-wrap: wrap` AND the ellipsis children need `min-width: 0` (flex children default to `min-width: auto` = content size, which defeats `text-overflow: ellipsis`). Both are now in the `@media (max-width: 600px)` block (`.corrections-list li .correction-change { flex-wrap: wrap }` + `.corrections-list li .correction-old/.correction-new { min-width: 0 }`); same specificity as the base `.history-list li …` rules but later-in-source so they win only at ≤600px. Reuse this two-level fix (stack the row AND unwrap+`min-width:0` any inner nowrap flex segment) for the next narrow-clipping row. Verify the whole chain at 420px with the `/tmp/sweep.js` overflow probe: `el.getBoundingClientRect().right > clientWidth` flags any element past the viewport edge even when `document.scrollWidth` stays clamped (an internal-`overflow` list hides the spill rather than scrolling, so doc-level checks miss it).
- **Fixed-height `overflow:auto` lists hide content with no affordance** (SOLVED for `.ledger-list`): the ledger/review block list uses the `background-attachment: local` scroll-shadow technique — two local-attachment cover gradients hide edge shadows when at scroll boundaries; two scroll-attachment radial shadows (`--scroll-shadow` token, light: rgba(0,0,0,0.10) / dark: rgba(255,255,255,0.07)) reveal when content extends beyond the visible area. Pattern: `background-color: var(--background)` must come AFTER the `background` shorthand in the same rule block (shorthand resets it to transparent; the longhand overrides). Apply the same technique to any future `overflow-y: auto` list that is viewport-height-capped.
- **Score fields are 0–1, not 0–100 — keep the demo seed on the same scale**: the live model emits `context_switch_score` and `wip_load_score` as 0–1 ratios (`Number(x.toFixed(2))`), and `RiskRow` expects 0–1 (it does `bounded * 100`). The `demoData.ts` `demoSnapshot` seed historically used 0–100 placeholders (44/52) for these — harmless while `snapshotHistory` was write-only, but once a consumer reads them (baseline chips) the cross-week deltas blow up. Fixed to 0.44/0.52 (+ history 0.38/0.49/0.45). Lesson: when seeding a snapshot/score field for demo, match the live computation's units exactly — `pct` fields are 0–100, `*_score` fields are 0–1.
- **Demo seed must exercise every flagship visual**: demo mode is the first thing a new user/evaluator sees, so a feature that renders blank in demo reads as broken. When a visual has a data threshold (e.g. `ActivityHeatmap` requires `daysWithActivity >= 2`), make the demo seed clear it — spread sessions across multiple days (fixed: `demoData.ts` now seeds 4 distinct days). General rule: walk every demo screen and confirm no flagship widget is in its empty/low-data fallback.
- **Center label positioned over a range track collides with edge labels near the ends** (SOLVED for `.forecast-range-label-center`): in `ForecastAgentPanel.tsx` the "Likely · X%" span is absolutely positioned at `left: ${likelyLeft}%` (translateX(-50%)) while the Conservative/Optimistic labels sit at the row edges (`:first-child`→left:0, `:last-child`→right:0). When `likelyLeft` skews near 0/100 the center label overlaps an edge label. Fix: render the center label only when `likelyLeft` is 12–88% (`showLikelyLabel`). The value stays visible via the summary card + range `aria-label`, and the marker line on the track still shows the position — so hiding the redundant text label loses no information. Conditionally omitting the middle span keeps the `:first-child`/`:last-child` CSS selectors correct in both paths. Reuse this hide-when-near-the-end approach for any future center-over-track label.
- **`auto-fit`/`auto-fill` grids wrap lopsided for fixed small counts**: a `repeat(auto-fit, minmax(240px, 1fr))` grid with a KNOWN fixed item count (e.g. `.forecast-grid`'s 4 `ForecastList`s) computes its column count from container width, not the item count — at a wide panel it packs 3 across and strands the 4th alone on a lopsided second row. For a fixed even count, prefer an explicit `repeat(2, minmax(0, 1fr))` (2×2) with a single-column `@media (max-width: 600px)` fallback. Reserve `auto-fit` for genuinely unbounded lists (ledger blocks, copilot suggestions).
