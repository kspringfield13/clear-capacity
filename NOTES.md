# NOTES.md — Architectural learnings for the ClearCapacity autonomous loop

Cross-run memory for the autonomous agents (hourly improver, 6-hour UX curator) and human readers.
Cloud sessions are stateless — without this file, every run relearns the codebase from scratch.

**Read this file right after STATUS.md. Append durable learnings here before you commit.**

## How to maintain this file
- Record only DURABLE, reusable insights: established patterns, conventions, gotchas, non-obvious decisions — things that save the next run time.
- Do NOT duplicate CLAUDE.md (build commands, repo layout) or STATUS.md (the task backlog). Refer to them instead.
- Keep it tight and curated (aim < ~150 lines). Update or delete entries that have gone stale; don't append forever.
- This is not a changelog — per-task "what I shipped" lines belong in CHANGELOG.md.

## Established UI patterns — reuse these, don't reinvent
- **Async op state** → use the `useAsyncStatus` hook (`apps/desktop/src/hooks/useAsyncStatus.ts`): `start()/fail()/reset()` instead of paired status+error `useState`. All five AI operations already use it.
- **Empty states** → route every screen's empty case through the shared `EmptyState` component (icon + headline + primary CTA). Don't write ad-hoc empty markup.
- **Inline error recovery** → wrap AI error text in the `.error-row` / `.error-retry` pattern with a "Try again" button wired to the existing retry callback.
- **Loading** → use the `skeleton-shimmer` animation + `.skeleton-line` / `.skeleton-block` utility classes so the layout doesn't shift when results arrive (don't show bare "Generating…" strings).
- **Per-screen toolbar action** → the primary action per screen is wired via the screen-aware `toolbarActions` logic in `App.tsx`.
- **Screen navigation** → ⌘1–⌘6 keyboard shortcuts live in `App.tsx`; respect the input/textarea focus guard when adding any new key handler.

## Conventions
- **Styling**: Vercel Geist design tokens via CSS variables in `apps/desktop/src/styles.css`. Support light AND dark. NO hardcoded hex — add a token with light/dark variants instead.
- **AI prompts**: the five prompt builders in `apps/desktop/src/services/*Prompt.ts` are version-stamped; bump the version string when you change a prompt's behavior.
- **Domain types**: shared types (`WorkBlock`, `ActivitySession`, `AIConfig`, `AuditEvent`, …) live in `packages/domain/src/models.ts`; capacity/grouping logic in `packages/inference/src/`.

## Gotchas
- **Render for visual checks**: the React UI runs in a plain browser via `npm run dev` (Vite, 127.0.0.1:5173); append `?demo=1&screen=<daily|weekly|narrative|ledger|audit|setup>` for synthetic data. Tauri APIs are absent in the browser — demo mode is how you render screens headless.
- **Headless dark mode**: the app does NOT honor `prefers-color-scheme`. Theme is `document.documentElement.dataset.theme`. Setting `localStorage["clear-capacity:theme"]` via `addInitScript` is NOT enough — `App.tsx` inits `theme` state to `"light"` and its mount effect overwrites both `dataset.theme` and the stored key back to light (see theme-persistence bug below). The reliable Playwright recipe: `goto` + wait, then `page.evaluate(() => { document.documentElement.dataset.theme = 'dark'; })` AFTER the app has mounted, then screenshot. (`colorScheme: 'dark'` alone does nothing.)
- **Theme flash on first paint (cosmetic)**: `main.tsx` sets `dataset.theme = "light"` synchronously then async-patches it; `App.tsx`'s mount effect briefly resets to "light" before hydration resolves. Cosmetic only — preference is now correctly persisted (see fix below).
- **Playwright in the sandbox**: the chromium CDN download is blocked, but a usable browser ships at `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` — pass it as `executablePath`. Install the `playwright` npm module under /tmp, never in the repo.
- **Icon-only button a11y convention**: chrome/icon buttons should carry both `title` and `aria-label` (plus `aria-pressed` for toggles). The theme toggle in `AppToolbar.tsx` is the reference; match it when adding icon controls.
- **Screen-reader live regions**: use `role="alert"` on a conditionally-rendered element (or a persistently-rendered element whose content changes) to announce validation errors to screen readers. The `.sr-only` utility class in `styles.css` (1×1px, clipped, absolute, `border: 0`) hides the element visually without removing it from the accessibility tree — use it for any visually-hidden but SR-meaningful text.
- **Verification gate**: `npm run build` (`tsc -b && vite build`) must pass before committing. There is no automated test suite.
- **Out of loop scope**: `apps/desktop/src-tauri/` (Rust), `.env`, and the 5173 port config are off-limits to the frontend loop. Anything needing Rust (model params, request bodies, JSON schemas, per-command model routing) is a manual follow-up — leave a clear note in STATUS.md.
- **Branch protection on main**: direct `git push origin main` returns HTTP 403. All loop commits must go via an `improve/<slug>` branch + PR (GitHub MCP `create_pull_request`). The STATUS.md "Never" section already requires this pattern.
- **npm install required in fresh containers**: `node_modules` is absent in new cloud sessions; run `npm install` before `npm run build` or the tsc step fails with "Cannot find module 'react'" errors on every file.
- **Stale local origin/main**: cloud sessions clone with a local `origin/main` that may lag behind GitHub. Always `git fetch origin main` at the start of a run before branching — otherwise `improve/*` branches will be based on an old commit and `git show origin/main:FILE` will differ from the working tree. If STATUS.md in the working tree doesn't match `git show HEAD:STATUS.md`, the repo was cloned at a stale commit; fetch + reset before proceeding.
- **STATUS.md corruption risk**: curator-pass PRs that update STATUS.md can silently revert it if the curator's branch was based on an older main. If STATUS.md on main looks stale (old Done items, no recent improvements), check `git log --oneline origin/main` against CHANGELOG.md — the CHANGELOG is the reliable source of truth for what's shipped. Reconstruct STATUS.md from it.
- **Verify the task is actually unimplemented before coding**: curator passes (and STATUS duplicates) routinely re-add items already fixed in code — a single run found the top 3 UI items already solved (agent-starters `flex-start`, capture panel already a closed `<details>`, current-block already picked by latest `end_time` via `reduce`). Always open the referenced file/lines FIRST and confirm the bug still exists; if already fixed, move it straight to Done with a "verified already solved" note and pick the next genuinely-open task instead of writing a redundant diff.

## Recurring UX rough edges (watch for these)
- **Don't render raw enum/snake_case values in the UI**: domain enums like `PrivacyLevel` (`local_only`/`derived_only`/`excluded`) and `PlannedStatus` (`planned`/`unplanned`/…) must pass through a humanizing helper in `lib/format.ts` (`auditTypeLabel`/`plannedStatusLabel`/`privacyLevelLabel`) before display — never `{event.privacy_level}` directly. Note `WorkCategory`/`WorkMode` are stored as already-human strings in `taxonomy.ts`. `CorrectionsScreen` and the `user_correction` audit-event summary in `App.tsx` both now humanize via `humanizeCorrectionValue(correction.field, …)` (fixed). Lesson: when a value is humanized for display, also humanize any pre-baked `summary`/`title` string stored on the audit event — humanize at event-construction time, not just at render. **This includes the demo seed**: `services/demoData.ts` hardcodes audit summaries directly (e.g. the `user_correction` event used a raw `"planned -> unplanned"`), so a seed string bypasses the live `addCorrection` humanizer and renders raw in demo mode while the Corrections screen shows it humanized. Demo mode is the showcase a new user/evaluator sees first — keep seed summaries consistent with the humanized live path (`lib/format.ts` helpers).
- **Group paired nav controls**: when chevrons/prev-next controls straddle a long headline (e.g. `.week-nav` in `WeeklyCapacityScreen`), they drift apart and the disabled one becomes invisible. Keep paired controls adjacent and give `:disabled` a visible low-opacity + `cursor: not-allowed` state.
- **Intensity-coded grids need a legend AND non-visual semantics**: `data-level` heatmap cells (`ActivityHeatmap`) need a "Less → More" key (done) plus per-cell `role="img"`+`aria-label` and a grid summary — hover `title` alone is invisible to keyboard/SR. They also look broken when sparse: the full 7×24 grid renders whenever there is ≥1 session, so a single active day reads as an empty box — add a low-data caption.
- **Numeric metric rows need scale + consistent units**: bare numbers like the `RiskRow` magnitudes ("33/90/10/46") read as ambiguous without a "/100" hint or a `title` tooltip. The raw-count-vs-index mismatch is now SOLVED in `RiskRow` and is the reference pattern: "Active blockers" passes `displayValue` (raw count, no `/100` hint), `dangerActive` (red bar + `.risk-blocker-count` number class), and a count-specific `aria-valuetext` — do NOT re-add this as a bug. Reuse the same trio for any future count-among-indices row. Severity-by-value tinting is also SOLVED: `RiskRow.tsx` now sets `data-severity="low|mid|high"` on the fill span (thresholds <34/34–66/≥67); `styles.css` has matching light + dark rules (low: muted blue-gray, mid: amber, high: orange); `dangerActive` row keeps its inline red override which beats any CSS rule. Apply the same `data-severity` pattern to any future value-coded bar. Same rule for chips and bare metric figures: a labeled percentage on a card needs a hover tooltip (`title`) stating *what it measures*, plus an `.sr-only` mirror so it isn't hover-only. `ConfidenceChip` is the SOLVED reference — it carries `title="…% classification confidence"` and renders 0 as "Unscored" (not "Needs review 0%"); don't re-add it. The Weekly `.summary-score` "Summary confidence" card (`WeeklyCapacityScreen.tsx`) and the BlockCard `.block-capacity` "% of week" figure now both carry `title` + `.sr-only` — SOLVED, do not re-add.
- **Color-only legend keys need hover crosslinks**: prefer bidirectional hover-highlight between legend rows and bar segments (dim non-hovered peers to ~0.3–0.35 opacity with a 0.12s transition) over relying on color alone. `categoryColors` palette now has no near-duplicate hues — keep it that way when adding categories.
- **Narrow-layout header actions must opt into the full-width media query**: the `@media (max-width: 600px)` block in `styles.css` full-widths `.primary-action`/`.secondary-action`/`.header-actions`/`.review-header-actions`/`.summary-score`/`.search-box` (verified clean at 420px on Daily + Narrative). Any NEW header-action wrapper class is an orphan until added — its buttons each take `width: 100%` in a non-wrapping flex row and the trailing one overflows off-screen. When adding a per-screen header action group, add its class to that rule and stack it (`flex-direction: column`).
- **Native `<select>` truncates its current value**: in equal-width grids like `.tag-grid` (`BlockCard.tsx`) a long option label is clipped at rest. The convention is to set `title={...}` on every select reflecting the *current* value (humanized for enum fields via `lib/format.ts`) so hover reveals it — but `title` is invisible to keyboard/touch, so for the longest field prefer giving its column more width over relying on the tooltip alone. Separately, `title` reflects the *current value* — it is NOT a field-name label, so a relabel select in a header-less row (the three in `BlockCard.tsx`) has no accessible name telling SR users which field they're editing; give each such select a static `aria-label` naming the field ("Work category" / "Planned status" / "Work mode") in addition to the value `title`.
- **Stacked-bar segments need a remainder + per-segment a11y label**: `StackedBar` only renders the colored category segments; when they sum to <100% the container background shows as an unexplained gray tail. Render an explicit "Unallocated / buffer" remainder span and give every segment an `aria-label` (hover `title` alone is invisible to keyboard/SR users). Same applies to `RiskRow` magnitude bars — use `role="meter"` + `aria-valuetext`.
- **Audit-type pills are keyed by a per-type CSS class**: `AuditEventRow` sets `className={`audit-badge ${event.type}`}`, so EVERY `AuditEventType` needs a matching `.audit-badge.<type>` rule in `styles.css` with a *tinted* background + colored text (light AND dark). The convention is one distinct hue per type (sky/violet/teal/amber/blue/green/red…); a colorless `var(--text)`-on-transparent rule (as `narrative_generation` currently has) reads as un-styled and breaks the set. When a new audit type is added, add its tinted badge rule the same way.
- **Fixed-height `overflow:auto` lists hide content with no affordance** (SOLVED for `.ledger-list`): the ledger/review block list uses the `background-attachment: local` scroll-shadow technique — two local-attachment cover gradients hide edge shadows when at scroll boundaries; two scroll-attachment radial shadows (`--scroll-shadow` token, light: rgba(0,0,0,0.10) / dark: rgba(255,255,255,0.07)) reveal when content extends beyond the visible area. Pattern: `background-color: var(--background)` must come AFTER the `background` shorthand in the same rule block (shorthand resets it to transparent; the longhand overrides). Apply the same technique to any future `overflow-y: auto` list that is viewport-height-capped.
- **Score fields are 0–1, not 0–100 — keep the demo seed on the same scale**: the live model emits `context_switch_score` and `wip_load_score` as 0–1 ratios (`Number(x.toFixed(2))`), and `RiskRow` expects 0–1 (it does `bounded * 100`). The `demoData.ts` `demoSnapshot` seed historically used 0–100 placeholders (44/52) for these — harmless while `snapshotHistory` was write-only, but once a consumer reads them (baseline chips) the cross-week deltas blow up. Fixed to 0.44/0.52 (+ history 0.38/0.49/0.45). Lesson: when seeding a snapshot/score field for demo, match the live computation's units exactly — `pct` fields are 0–100, `*_score` fields are 0–1.
- **Demo seed must exercise every flagship visual**: demo mode is the first thing a new user/evaluator sees, so a feature that renders blank in demo reads as broken. When a visual has a data threshold (e.g. `ActivityHeatmap` requires `daysWithActivity >= 2`), make the demo seed clear it — spread sessions across multiple days (fixed: `demoData.ts` now seeds 4 distinct days). General rule: walk every demo screen and confirm no flagship widget is in its empty/low-data fallback.
- **Center label positioned over a range track collides with edge labels near the ends** (SOLVED for `.forecast-range-label-center`): in `ForecastAgentPanel.tsx` the "Likely · X%" span is absolutely positioned at `left: ${likelyLeft}%` (translateX(-50%)) while the Conservative/Optimistic labels sit at the row edges (`:first-child`→left:0, `:last-child`→right:0). When `likelyLeft` skews near 0/100 the center label overlaps an edge label. Fix: render the center label only when `likelyLeft` is 12–88% (`showLikelyLabel`). The value stays visible via the summary card + range `aria-label`, and the marker line on the track still shows the position — so hiding the redundant text label loses no information. Conditionally omitting the middle span keeps the `:first-child`/`:last-child` CSS selectors correct in both paths. Reuse this hide-when-near-the-end approach for any future center-over-track label.
- **`auto-fit`/`auto-fill` grids wrap lopsided for fixed small counts**: a `repeat(auto-fit, minmax(240px, 1fr))` grid with a KNOWN fixed item count (e.g. `.forecast-grid`'s 4 `ForecastList`s) computes its column count from container width, not the item count — at a wide panel it packs 3 across and strands the 4th alone on a lopsided second row. For a fixed even count, prefer an explicit `repeat(2, minmax(0, 1fr))` (2×2) with a single-column `@media (max-width: 600px)` fallback. Reserve `auto-fit` for genuinely unbounded lists (ledger blocks, copilot suggestions). Watch the small-`minmax` variant too: `.forecast-summary` uses `minmax(min(170px, 100%), 1fr)`, which still packs TWO cramped ~190px columns at a 420px viewport — fixed-count card rows usually want an explicit `grid-template-columns: 1fr` in the narrow media query so they stack instead of squeezing.
- **Compact media queries that `display: none` real content (anti-pattern — audit when adding narrow rules)**: the `@media (max-width: 600px)` block in `styles.css` currently HIDES content rather than reflowing it — `.briefing-metrics > div:last-child` drops the Agent "Outlook / carryover risk" metric, and `.starter-grid > button:nth-child(n + 3)` hides 2 of the 4 Agent "Start with an outcome" cards on phones. This is the same "No silent caps" failure as truncating a list without saying so: a phone user loses information with no signal. Prefer stacking (single-column) over hiding for any meaningful metric/card; reserve `display: none` for genuinely redundant chrome. (Both are tracked under STATUS → UI & UX Polish.)
- **Empty-state layout: anchor, don't center into a void** (SOLVED): `.agent-starters` now uses `justify-content: flex-start` so the "Start with an outcome" cards sit just below the briefing card. Pattern: for empty states above a fixed composer, always use `flex-start` not `center` — centering into a tall flex container creates large dead bands that read as broken.
- **Absolutely-positioned marker labels collide with edge labels near the ends** (SOLVED in `ForecastAgentPanel.tsx`): a label positioned at `left: ${pos}%` + `translateX(-50%)` overlaps an edge-anchored (`left:0`/`right:0`) label when `pos` is near 0 or 100. Convention: gate it with a threshold (e.g. render only when `pos > 12 && pos < 88`) and rely on the value being duplicated elsewhere (summary card / container `aria-label`) so hiding loses no info. Conditional render keeps CSS `:first-child`/`:last-child` selectors valid since the center span never matches them.
- **Stacked label+subtitle needs a block/column wrapper, not two bare inline spans** (SOLVED for `ActivityCapturePanel`): a title span followed by a subtitle span inside a plain `<div>` (no flex/gap, no whitespace between them) renders run-together on one line. Fixed in `ActivityCapturePanel.tsx`'s `<summary>` by wrapping the pair in `.capture-panel-heading` (`display: flex; flex-direction: column; gap: 2px; min-width: 0`) and giving the subtitle `.capture-panel-subtitle` (`var(--text-subtle)`, 12px), mirroring the panel's own `.capture-stat` (span/strong/small stacked). Reuse this column-wrapper convention for any future title+subtitle pair; remember `min-width: 0` on the heading so it shrinks instead of pushing sibling actions off-screen at 420px.
- **`<details>` disclosures must keep a visible expand affordance** (SOLVED reference for `.activity-capture-panel`): the heatmap/evidence panels keep the browser's default ▶ marker so they read as expandable; `.activity-capture-panel summary::-webkit-details-marker { display: none }` strips it. The fix (don't drop the hide rule — substitute an explicit caret): render a `<ChevronRight aria-hidden size={16} className="capture-panel-caret" />` as the first child of a `.capture-panel-summary-main` flex wrapper (`align-items: center; gap: 8px; min-width: 0`) that holds the caret + the heading, so the caret stays adjacent to the title while the right-side action group keeps the summary's `space-between`. Caret is decorative (`aria-hidden`; the summary text supplies the accessible name), colored `var(--text-subtle)`, and rotates via `.activity-capture-panel[open] .capture-panel-caret { transform: rotate(90deg) }` with a 0.15s transition. Reuse this caret-in-a-flex-wrapper pattern for any future `<details>` whose native marker is hidden — never leave a collapsible region with zero affordance.
- **Audit privacy-scope pills ARE differentiated (don't re-flag)**: `.audit-privacy--local_only` (green) vs `.audit-privacy--derived_only` (blue) vs `.audit-privacy--excluded` (muted) each have distinct light + dark rules in `styles.css`. At thumbnail scale the dark-mode green/blue can look similar, but they are correctly distinct — verify at full resolution before treating "all the pills look the same" as a bug.

## Open architectural notes
- `App.tsx` is a state-wiring orchestrator (~828 lines). Decomposition complete: AI ops in `hooks/use*.ts`, toolbar actions in `lib/toolbarActions.ts` (`buildToolbarActions`), screen routing in `components/shell/ScreenRouter.tsx`. Future growth should land in new hooks or components, not App.tsx.
- **Async op hook pattern**: each AI hook takes a params object (data + setter callbacks), calls `useAsyncStatus`, defines the async function + any internal `useEffect` triggers, and returns `{status, error, run, reset}`. The auto-trigger effects (`useNarrativeGeneration`, `useVisualContext`) live inside the hook. `resetLocalData` in App.tsx calls each hook's `reset` function.
- **Hook ordering in App.tsx**: the AI hooks are called AFTER `useDerived` because they need `snapshot`, `activeWindowSessions`, `todayKey`, `hasNarrativeEvidence` from `useDerived`. This ordering does not violate Rules of Hooks (all calls are unconditional).
- **Retained-history persistence pattern** (reuse for any cross-week/cross-run store): mirror `forecastHistory` / `snapshotHistory`. (1) Define a `Persisted*Record` type + add the array field to `PersistedAppState` in `services/localStore.ts`; (2) add a `parse*History` guard and call it in BOTH `readPersistedState` branches (Tauri store + localStorage fallback) — missing one branch silently drops the field for that environment; (3) add the field to `PersistableState` AND its effect dep array in `hooks/usePersistence.ts`; (4) in `App.tsx` add the `useState`, hydrate it in both the async-load effect and the lazy initializer, clear it in `resetLocalData`, and pass it to `usePersistence`; (5) seed it in `services/demoData.ts` (required field — TS will fail the build otherwise). `snapshotHistory` retains one snapshot per ISO `week_id` via an upsert `useEffect` keyed on the live `snapshot` (latest wins, cap 24, `JSON.stringify`-equality dedup to avoid write churn, guarded by `!isDemoMode && blocks.length > 0`). Its first consumer is now wired: `App.tsx` → `ScreenRouter` → `WeeklyCapacityScreen`, which feeds the weeks strictly before the viewed `week_id` into the pure `computeCapacityBaselines(history)` (`packages/inference/src/capacity.ts`) and renders "vs your N-wk median" delta chips. Reuse that read path (pass `snapshotHistory` records as a prop, `.filter(week_id < target).map(r => r.snapshot)`, then a pure inference helper) for the other history-backed tasks rather than threading through `useDerived`.
