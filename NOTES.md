## Established UI patterns — reuse these, don't reinvent
- **Async op state** → use the `useAsyncStatus` hook (`apps/desktop/src/hooks/useAsyncStatus.ts`): `start()/fail()/reset()` instead of paired status+error `useState`. All five AI operations already use it.
- **Empty states** → route every screen's empty case through the shared `EmptyState` component (icon + headline + primary CTA). Don't write ad-hoc empty markup; its `children` slot renders into `.empty-state-actions` (already full-widths/stacks at ≤600px) — put navigation buttons (`primary-action`/`secondary-action`) there. The empty `daily`/`weekly`/`forecast` screens all use "Import calendar"/"Open Settings" buttons → `onOpenScreen("setup")`.
- **Onboarding checklist** → `buildOnboardingSteps(status)` + `<OnboardingCard>` (`components/common/OnboardingCard.tsx`) are the single source for the getting-started steps (track → import → configure AI → classify). Both `SetupScreen` (inline, `!allDone`) and the first-run card on empty daily/weekly render it; the card takes an optional `onDismiss` (renders the `X`). First-run gating lives in `App.tsx` as `showOnboarding = !isDemoMode && !onboardingDismissed && blocks.length === 0`, threaded through `ScreenRouter`. Don't re-inline the step list anywhere — extend the helper.
- **Inline error recovery** → use the shared `components/common/InlineError.tsx` (`<InlineError message={err} onRetry={handler} />`) — a single `role="alert"` `.error-row` with an `AlertTriangle` icon + `.inline-error-text` box + optional "Try again" button. This is the ONE source for persistent in-panel AI-op errors (ReviewCopilot/Forecast/Narrative/classification all route through it); don't hand-roll an `.error-row`/`.error-retry` block. Style lives in `.inline-error-text` (light/dark) — the old per-panel `.copilot-error`/`.forecast-error`/`.narrative-error` classes were removed. Gotcha: `.narrative-screen` uses `gap: 0`, so it needs a scoped `.narrative-screen .error-row { margin-bottom: 14px }` rule to keep spacing (other panels get spacing from their parent's grid `gap`). Passive informational errors with no idempotent retry (capture/visual-context) stay as bare `.capture-error` paragraphs — the boxed InlineError style is reserved for *actionable* (retry-able) errors. Complementary to toasts (transient), not a substitute — don't double-render both for one event.
- **Loading** → use the `skeleton-shimmer` animation + `.skeleton-line` / `.skeleton-block` utility classes so the layout doesn't shift when results arrive (don't show bare "Generating…" strings).
- **Per-screen toolbar action** → the primary action per screen is wired via the screen-aware `toolbarActions` logic in `App.tsx`.
- **Confirm/alert dialogs** → use the shared `components/common/ConfirmDialog.tsx` for any destructive/irreversible confirm — don't hand-roll another modal. It is a `role="alertdialog"` + `aria-modal` panel with the loop's a11y baseline already built in: `useId`-keyed `aria-labelledby`/`aria-describedby`, a Tab/Shift+Tab focus trap, Esc + overlay-click (`onMouseDown` target===currentTarget) cancel, initial focus on the safe **Cancel** control (not the destructive confirm), and focus restore to the trigger via a mount-time `document.activeElement` capture restored in the effect cleanup. Props: `title`/`description`/`confirmLabel`/`cancelLabel`/`tone` (`"danger"` → `.dialog-confirm-danger` white-on-`--danger`)/`onConfirm`/`onCancel` + a `children` slot. Render it conditionally from a local `useState` boolean; the trigger restores focus automatically when it unmounts. Styling is `.dialog-*` (overlay/panel/title/desc/delete-list/export-action/actions) — token-based, 420px-safe (`flex-wrap` actions), fade-in auto-zeroed by the global `prefers-reduced-motion` reset (no per-component guard needed).
- **Screen navigation** → ⌘1–⌘9 keyboard shortcuts (`SCREEN_KEYS` in `App.tsx`: 1 daily … 9 setup) live in `App.tsx`; respect the input/textarea focus guard when adding any new key handler. **Discoverable in the sidebar (SOLVED — don't re-flag "shortcuts are invisible"):** each `.nav-item` that maps to a `SCREEN_KEYS` entry (Today ⌘1, Week ⌘2, History ⌘5, Settings ⌘9) + the bottom `.settings-button` carry `title="Label (⌘N)"` + `aria-keyshortcuts="Meta+N"` (`AppShell.tsx`); Agent has no `SCREEN_KEYS` entry so its item is hint-free (explicit `shortcut: undefined`/`shortcutKey: undefined` keeps the items array's element type uniform). If you add a new ⌘-number, mirror it onto its nav item the same way.
- **Transient feedback (toasts)** → use the `useToasts` hook (`hooks/useToasts.ts`) + `<ToastHost>` (`components/common/ToastHost.tsx`) for app-level success/error/info that should auto-expire — NOT for persistent in-panel errors (those use the inline `.error-row`/`error-retry` pattern; the two are complementary, don't double-render the same Retry). The queue is owned in `App.tsx` (single source of truth); `pushToast` (stable `useCallback`, safe in effect deps) threads down through `ScreenRouter` props, and the visual stack renders once in `AppShell` (`toasts`/`onDismissToast`). To surface a swallowed AI error as a toast, add an effect keyed on the error string with a `prev*Error` ref guard (`error && error !== prev.current` → push; always write `prev.current = error`) so an unchanged value never re-announces but a genuine new failure fires once — `useAsyncStatus.start()` nulls the error before `fail()`, which makes the ref toggle work. `ToastInput`/`PushToast`/`Toast` types are exported from the hook. Stack caps at 4 (clear the evicted timer when slicing). Slide-in is auto-zeroed by the global reduced-motion reset — no per-component guard.

## Conventions
- **Styling**: Vercel Geist design tokens via CSS variables in `apps/desktop/src/styles.css`. Support light AND dark. NO hardcoded hex — add a token with light/dark variants instead.
- **AI-provider form help is preset-driven**: per-provider inline help in the Advanced Settings form (`SetupScreen.tsx`) lives ENTIRELY on the `AIProviderPreset` (`services/aiProviders.ts`), never hardcoded in the component — `modelNote`/`baseUrlNote`/`visionNote` (helper `<small>` under each field, styled by the shared `.ai-field > small` rule), `modelSuggestions?: string[]` (click-to-fill "RECOMMENDED" chips under Model; render falls back to `[model]`, empty for custom; chip `aria-pressed={draftConfig.model === id}` is the non-color selection cue), and `docsUrl?: string` (external "{label} model docs" link, `--info` blue, `ExternalLink` + `target=_blank rel="noopener noreferrer"`). To add help for a new provider/field, extend the preset object — don't add JSX. Header gotcha: the `.ai-provider-header` is `justify-content: space-between`, so the docs link + the "stored locally" note must be wrapped together in `.ai-provider-meta` or `space-between` strands the note in the center.
- **AI prompts**: the five prompt builders in `apps/desktop/src/services/*Prompt.ts` are version-stamped; bump the version string when you change a prompt's behavior.
- **Domain types**: shared types (`WorkBlock`, `ActivitySession`, `AIConfig`, `AuditEvent`, …) live in `packages/domain/src/models.ts`; capacity/grouping logic in `packages/inference/src/`.

## Gotchas
- **Render for visual checks**: the React UI runs in a plain browser via `npm run dev` (Vite, 127.0.0.1:5173); append `?demo=1&screen=<daily|weekly|narrative|ledger|audit|setup>` for synthetic data. Tauri APIs are absent in the browser — demo mode is how you render screens headless. **fullPage screenshots lie about internal-scroll screens**: the app is a fixed-viewport shell (the macOS-window model), so at every width EVERY content `.screen` is its own `overflow-y:auto` box capped at the remaining shell height (~500px at 420px wide, after the reflowed top nav) while holding far more content (a 420px DOM probe found weekly scrollHeight 2100, ledger 3398, settings 3200). Playwright `fullPage:true` cannot capture content hidden inside an inner `overflow` box, so those screens render as "a few cards then a huge blank band" — a SCREENSHOT ARTIFACT, not a truncation bug (the scroll-shadow affordance already cues the hidden content). To see below the fold, screenshot the `.screen` *element* (`el.screenshot()`) or scroll it; never file a "content cut off on narrow" item from a fullPage shot alone.
- **Headless dark mode**: the app does NOT honor `prefers-color-scheme`. Theme is `document.documentElement.dataset.theme`. Setting `localStorage["clear-capacity:theme"]` via `addInitScript` is NOT enough — `App.tsx` inits `theme` state to `"light"` and its mount effect overwrites both `dataset.theme` and the stored key back to light (see theme-persistence bug below). The reliable Playwright recipe: `goto` + wait, then `page.evaluate(() => { document.documentElement.dataset.theme = 'dark'; })` AFTER the app has mounted, then screenshot. (`colorScheme: 'dark'` alone does nothing.)
- **Theme flash on first paint (cosmetic)**: `main.tsx` sets `dataset.theme = "light"` synchronously then async-patches it; `App.tsx`'s mount effect briefly resets to "light" before hydration resolves. Cosmetic only — preference is now correctly persisted (see fix below).
- **Playwright in the sandbox**: the chromium CDN download is blocked, but a usable browser ships at `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` — pass it as `executablePath`. Install the `playwright` npm module under /tmp, never in the repo.
- **Icon-only button a11y convention**: chrome/icon buttons should carry both `title` and `aria-label` (plus `aria-pressed` for toggles). The theme toggle in `AppToolbar.tsx` is the reference; match it when adding icon controls. **If the button swaps its ICON to signal a transient state (e.g. the Agent copy button → `Check` on `copiedMessageId === m.id`), mirror that same predicate into the `title` AND `aria-label` too** — the icon swap is invisible to screen-reader/tooltip users, so a static accessible name silently drops the success confirmation. `AgentScreen.tsx`'s copy button is the reference (`copiedMessageId === m.id ? "Copied" : "Copy response"` on both attrs).
- **Screen-reader live regions**: use `role="alert"` on a conditionally-rendered element (or a persistently-rendered element whose content changes) to announce validation errors to screen readers. The `.sr-only` utility class in `styles.css` (1×1px, clipped, absolute, `border: 0`) hides the element visually without removing it from the accessibility tree — use it for any visually-hidden but SR-meaningful text.
- **Don't change a persisted source's `event_id` format** (idempotency): an imported block's `work_block_id = imported-${stableHash(event_id)}`, and `importWorkplaceChat`/calendar upsert by `work_block_id`. Reformatting an existing source's `event_id` (e.g. adding a prefix to ALL chat bursts) re-hashes every previously-persisted block → re-import mints NEW ids that don't collide with the stored ones → duplicates that double-count `*_pct`. Rule: keep the historical id format byte-stable; only NEW variants take a new prefix. Worked example: chat call bursts use `chat-call-<provider>-<iso>` but text bursts kept the original `chat-<provider>-<iso>` so a pre-existing export still upserts.
- **Cross-source block dedup keys off `work_block_id` prefixes**: calendar blocks are `calendar-outlook-…` (from `outlookEventsToWorkBlocks`, always category `Meetings / stakeholder syncs`); every `importRawEvents` block (chat/git/browser/…) is `imported-…`. Use these prefixes to tell origins apart when merging/deduping in `App.tsx` (the calendar handler already filters `!startsWith("calendar-outlook-")`). **Import-time dedup must be symmetric** — a feature that only dedups source A against source B at A's import silently fails when B is imported after A. The chat call-vs-calendar dedup (`chat/callDedup.ts#dedupeChatCallsAgainstCalendar`, pure, half-open `[start,end)` `spansOverlap`) runs in BOTH handlers: chat-import drops new call blocks overlapping current calendar meetings; calendar-import drops existing `imported-` call blocks overlapping the new calendar meetings. Caveat (pre-existing, not unique to this): re-import upserts by id and overwrites user edits to imported blocks. **Now also surfaced in the UI** — `BlockCard.tsx#blockOrigin(work_block_id)` maps these same prefixes to a muted `.block-origin` provenance chip (`calendar-outlook-`→"Calendar", `imported-`→"Workplace chat", else→"Activity capture"); capture blocks use `ai-session-`/`session-` ids and correctly fall through to the default. **Demo gotcha when keying UI off the prefix**: `demoData.ts` work_block_ids are synthetic (`demo-*`) and do NOT follow the production prefix convention, so a feature that reads the prefix mislabels demo blocks until the seed ids are aligned — the two demo calendar meetings were renamed to `calendar-outlook-demo-*` so the chip reads "Calendar". When adding any prefix-keyed UI, audit the demo seed ids too (demo is the first thing an evaluator sees).
- **Verification gate**: `npm run build` (`tsc -b && vite build`) must pass before committing. There is no automated test suite.
- **Out of loop scope**: `apps/desktop/src-tauri/` (Rust), `.env`, and the 5173 port config are off-limits to the frontend loop. Anything needing Rust (model params, request bodies, JSON schemas, per-command model routing) is a manual follow-up — leave a clear note in STATUS.md.
- **Branch protection on main**: direct `git push origin main` returns HTTP 403. All loop commits must go via an `improve/<slug>` branch + PR (GitHub MCP `create_pull_request`). The STATUS.md "Never" section already requires this pattern.
- **npm install required in fresh containers**: `node_modules` is absent in new cloud sessions; run `npm install` before `npm run build` or the tsc step fails with "Cannot find module 'react'" errors on every file.
- **Stale local origin/main**: cloud sessions clone with a local `origin/main` that may lag behind GitHub. Always `git fetch origin main` at the start of a run before branching — otherwise `improve/*` branches will be based on an old commit and `git show origin/main:FILE` will differ from the working tree. If STATUS.md in the working tree doesn't match `git show HEAD:STATUS.md`, the repo was cloned at a stale commit; fetch + reset before proceeding.
- **STATUS.md corruption risk**: curator-pass PRs that update STATUS.md can silently revert it if the curator's branch was based on an older main. If STATUS.md on main looks stale (old Done items, no recent improvements), check `git log --oneline origin/main` against CHANGELOG.md — the CHANGELOG is the reliable source of truth for what's shipped. Reconstruct STATUS.md from it.
- **Verify the task is actually unimplemented before coding**: curator passes (and STATUS duplicates) routinely re-add items already fixed in code — a single run found the top 3 UI items already solved (agent-starters `flex-start`, capture panel already a closed `<details>`, current-block already picked by latest `end_time` via `reduce`). Always open the referenced file/lines FIRST and confirm the bug still exists; if already fixed, move it straight to Done with a "verified already solved" note and pick the next genuinely-open task instead of writing a redundant diff.

- **Don't audit-log high-frequency background effects** (SOLVED for retention auto-expiry): the audit trail is capped at `.slice(-1000)`, and `activeWindowSamples` accrue one-per-native-event, so an effect that logs on every sample change (e.g. logging each retention expiry as samples continuously age past a cutoff) floods the trail and evicts genuinely meaningful events (classifications, corrections). Rule: log the DISCRETE user action that sets the policy (e.g. `changeRetentionDays` → one `retention_policy` event per window change), NOT the recurring background consequence. Auto-expiry/cleanup effects should mutate state silently. Pattern for the effect itself: filter + `return kept.length === current.length ? current : kept` inside the functional update so it returns the SAME reference when nothing crosses the boundary (no re-render loop even though the effect deps include the array it mutates), and early-return in demo mode.

- **Week section has FOUR sub-tabs now (Capacity / Forecast / Trends / Summary)** — the `Trends` screen (`components/capacity/TrendsScreen.tsx`) is a multi-week line chart over `snapshotHistory` (allocated/reactive/deep-work/reliable, pure SVG, `role="img"` + per-dot `<title>`, labeled `.trend-legend`) plus the forecast track record below it. **Panel-home convention (SOLVED — don't re-flag):** `ForecastTrackRecord` now lives ONLY on `TrendsScreen` (the per-week predicted-vs-actual history pairs with the multi-week chart); it was removed from `ForecastScreen`, which keeps the `ForecastAgentPanel` accuracy banner + trend line (those are about *next* week). `forecastTrackRecord` is still a `ScreenRouterProps` field but is passed to the Trends call site only — when adding a history-flavored panel, the Trends tab is its home; don't also render it on Forecast. Also: the Trends subtitle and `ForecastAgentPanel`'s accuracy line both surface the same `ForecastAccuracyTrend` — keep their copy consistent ("over the last N weeks", with a singular guard).

- **The Review Copilot suggestion card lags the app's display conventions** (`components/review/ReviewCopilotPanel.tsx`): two of the three 2026-06-30-visual-pass gaps are now SOLVED — (1) it now humanizes the action via `reviewActionLabel(suggestion.action)` (`lib/format.ts`, exhaustive `Record<ReviewCopilotAction,string>` mirroring `plannedStatusLabel`; "note"→"Add note", "merge"→"Merge blocks", "split"→"Split block") instead of the raw lowercase enum — don't re-flag "raw enum"; (2) its `· N%` confidence now carries a `title` + explanation-only `.sr-only` gloss, so the metric title+sr-only sweep is genuinely complete again. The ONE remaining gap: (3) it cites no block by name, only a bare "N block(s)" count, unlike every other evidence-citing surface. Panel receives only `suggestions` today — naming the affected block needs a `work_block_id → title` map threaded from `DailyReviewScreen.tsx`.
- **Group paired nav controls**: when chevrons/prev-next controls straddle a long headline (e.g. `.week-nav` in `WeeklyCapacityScreen`), they drift apart and the disabled one becomes invisible. Keep paired controls adjacent and give `:disabled` a visible low-opacity + `cursor: not-allowed` state.
- **Intensity-coded grids need a legend AND non-visual semantics**: `data-level` heatmap cells (`ActivityHeatmap`) need a "Less → More" key (done) plus per-cell `role="img"`+`aria-label` and a grid summary — hover `title` alone is invisible to keyboard/SR. They also look broken when sparse: the full 7×24 grid renders whenever there is ≥1 session, so a single active day reads as an empty box — add a low-data caption.
- **Numeric metric rows need scale + consistent units**: bare numbers like the `RiskRow` magnitudes ("33/90/10/46") read as ambiguous without a "/100" hint or a `title` tooltip. The raw-count-vs-index mismatch is now SOLVED in `RiskRow` and is the reference pattern: "Active blockers" passes `displayValue` (raw count, no `/100` hint), `dangerActive` (red bar + `.risk-blocker-count` number class), and a count-specific `aria-valuetext` — do NOT re-add this as a bug. Reuse the same trio for any future count-among-indices row. Severity-by-value tinting is also SOLVED: `RiskRow.tsx` now sets `data-severity="low|mid|high"` on the fill span (thresholds <34/34–66/≥67); `styles.css` has matching light + dark rules (low: muted blue-gray, mid: amber, high: orange); `dangerActive` row keeps its inline red override which beats any CSS rule. Apply the same `data-severity` pattern to any future value-coded bar. Same rule for chips and bare metric figures: a labeled percentage on a card needs a hover tooltip (`title`) stating *what it measures*, plus an `.sr-only` mirror so it isn't hover-only. `ConfidenceChip` is the SOLVED reference — it carries `title="…% classification confidence"` and renders 0 as "Unscored" (not "Needs review 0%"); don't re-add it. The Weekly `.summary-score` "Summary confidence" card (`WeeklyCapacityScreen.tsx`), the BlockCard `.block-capacity` "% of week" figure, the four Weekly `.interruption-stat` tiles ("reactive bursts" / "messages while active" / "direct @-mentions" / "deep work interrupted"), and the three Agent `.briefing-metrics` tiles ("Planned" / "Reactive" / "Outlook", `AgentScreen.tsx`) now all carry `title` + `.sr-only` — SOLVED, do not re-add. The always-visible sidebar `.side-metric-value` "Reliable capacity" number in `AppShell.tsx` is ALSO now SOLVED (title "Estimated share of next week that can absorb new planned work without slippage" + explanation-only `.sr-only`) — that was the last holdout, so the metric title+`.sr-only` convention sweep is now COMPLETE across the app; verify in the named file before adding any "metric lacks tooltip/sr-only" task. **The shared `MetricCard` (`components/common/MetricCard.tsx`) now takes an optional `title?: string` prop** that renders the gloss as a hover tooltip on the `.metric-card` container AND an explanation-only `.sr-only` span (defaults `undefined` → no attr + no span, so cards that omit it are unchanged) — used on the Weekly "Reliable new work" hero card to decode the "80% knee" queueing-theory jargon. Pass `title` (not a hand-rolled `<small title>`) for any future MetricCard whose `helper` carries jargon. **`.sr-only` mirror convention:** make the hidden span EXPLANATION-ONLY (what the metric means), NOT a restatement of the visible `<strong>` value + `<span>` label — the visible value+noun is already in the a11y tree, so a value-repeating mirror double-announces (e.g. "3 reactive bursts … 3 reactive bursts — a burst is…"). `.summary-score`'s sr-only is the reference (pure explanation). Only repeat a value when the mirror adds info the visible figure omits (the baseline-chips restate share% + add `burst_count`; the interruption deep-work tile adds the "{interrupted} of {total} blocks" breakdown the visible `%` doesn't show).
- **Top-N truncated rows must surface the hidden count** (SOLVED for the stakeholder chips — don't re-flag): a list/chip row that silently `.slice(0, limit)`s its data misleads when the shares/parts no longer sum to the whole. `summarizeChatStakeholders` already computes `group_count` (distinct groups *before* the top-N cut) alongside the sliced `groups` — `WeeklyCapacityScreen.tsx` now renders a `.stakeholder-chips-note` footnote ("Top N of M groups by reactive volume — K more not shown") gated on `group_count > groups.length`. Pattern: when you `.slice` a ranked list for display, keep the pre-slice count on the summary type and render a "+K more"/"top N of M" caption so the truncation (and any sub-100% share total) is explicit. The footnote uses `flex-basis:100%` to drop onto its own line inside the `.baseline-chips` flex row; `.capacity-caption` is NOT a clean reuse (it is `text-align:right`).
- **A count-gated footnote that also shows a rounded percentage must floor the percentage to 1**: when a caption is gated on `someCount > 0` but its text renders `Math.round(someCount / total * 100)`, a small count amid large volume rounds to 0 and prints a self-contradictory "0% (1 of 300)". Floor it: `count > 0 ? Math.max(1, Math.round(...)) : 0`. Worked example — the chat-interruption **after-hours footnote** (`WeeklyCapacityScreen.tsx`, gated on `after_hours_message_count > 0`) reads `after_hours_pct`, which `analyzeInterruptionLoad` (`capacity.ts`) floors to 1 for exactly this reason. Don't re-add this signal.) Reusable demo trick: to make such a footnote render without disturbing tuned peak/calm/stakeholder demo values, time-SHIFT an existing `demoData.ts` chat event (keep its day/channel/volume) rather than ADD one — shifting only the minute leaves every per-weekday and per-channel aggregate identical.
- **Color-only legend keys need hover crosslinks** (SOLVED across all multi-series viz — don't re-flag): prefer bidirectional hover-highlight between legend rows and bar segments/lines (dim non-hovered peers to ~0.3 opacity with a 0.12s transition) over relying on color alone. (1) A non-interactive status glyph rendered as `<button type="button">` with no `onClick` (e.g. the Agent composer `.agent-attach` paperclip — `title` "Context is attached automatically") still gets pointer cursor + focus ring + tab stop, so users click it expecting an action and nothing happens. Use a `<span>` + `aria-label` for decorative/status glyphs; reserve `<button>` for controls that DO something. **SOLVED** for `.agent-attach`: it is now a `<span role="img">` (out of the tab order, no hover/focus ring; the `:hover` rule was deleted) keeping its `aria-label` as the accessible name.
