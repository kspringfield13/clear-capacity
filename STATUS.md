### Workplace Chat & Collaboration Signals
_New direction: turn workplace chat (Slack, Microsoft Teams, Webex) into reviewable workload signals. Chat is the one source that exposes interruption load, ad-hoc/reactive work, and stakeholder mix — the parts of the capacity model calendar + git can't see. **Hard constraint:** message bodies are sensitive like window titles — this whole family is **metadata-only** (timestamps, channel/DM, direction, mention flag, thread id, participant counts), never message text, in evidence or any network call. Mirror two proven patterns: the `gitLog.ts` burst-sessionizer (commits→sessions via `importRawEvents`) and the `calendarSource.ts` provider-registry + `loopSafe` flags + documented Rust follow-up. **Use a single generic `chat` `SourceType`** (generalize the reserved `slack` entry) rather than one per vendor — most orgs standardize on a single chat app, so the user picks their provider once and the workload signal is identical across Slack/Teams/Webex; the specific app rides as a `provider` field on event metadata + the `ChatSource` descriptor, not as a separate source type. Most of this is "data, not code."_
