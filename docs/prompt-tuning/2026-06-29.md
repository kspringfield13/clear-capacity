## Finding A — Backlog drains faster than the curator replenishes → `## Next` is currently EMPTY (highest leverage)

### Observations
- `grep -c '^- \[ \]' STATUS.md` returns **0**. The three `## Next` subsections
  (Interaction & Visual Polish, Intelligence Engine, Trust & Verification UX) contain
  only italic reference preambles — **no `- [ ]` items at all**. The Interaction &
  Visual Polish preamble even advertises "_New batch from the 6-hour visual UX pass
  (2026-06-29)_" but is followed by zero items.
- Root cause, from the commit record: the last curator pass `#127` (`d1d50ae`) added
  exactly **3** `- [ ]` items. The next four hourly improver runs consumed them:
  `#128` (scroll-shadow), `#129` (BlockCard labels), `#130` (interruption tiles) —
  the 3 curator items — plus `#131`. Backlog reached zero ~4 hours after the curator pass.
- Cadence math: the improver runs **hourly**; the curator runs **every 6 hours**.
  So ~6 items are consumed between curator passes, but `#127` replenished only 3.
  Net drain → guaranteed periodic empty-backlog windows.

### Diagnosis
Two compounding prompt weaknesses:
1. The curator's replenish step (task B.3) gives no minimum count tied to the
   consumption rate, and the "Rules for items you add" section actively says
   *"prefer a few high-value … items"* — which under-fills relative to the 6
   items the improver will eat before the next curator pass.
2. The improver prompt has **no instruction for an empty `## Next`**. Step 2 says
   "Pick the top-most unchecked item under ## Next"; with none present the run's
   behavior is undefined (idle, fail, or invent an unvetted task).

### Proposed edit (curator — replenish buffer)
*Current* (PROMPTS.md task B.3, line ~70):
> 3. Replenish when depleted. If most ## Next work is done (or the UI/UX Polish subsection is thin), add new concrete bullet points derived from your visual UX audit and the app's current state. Bias heavily toward UI/UX polish …

*Replacement:*
> 3. Replenish to outlast the next inter-pass window. The hourly improver consumes ~6 items before you run again, so whenever fewer than ~6 `- [ ]` items remain under `## Next` (count them), add concrete bullet points until at least **6–8** are queued — derived from your visual UX audit and the app's current state. Never leave a subsection with a descriptive preamble but zero `- [ ]` items. Bias heavily toward UI/UX polish …

### Proposed edit (curator — relax the anti-bloat cap when depleted)
*Current* ("Rules for items you add", line ~81):
> - Don't bloat the list — prefer a few high-value, well-scoped items over many vague ones.

*Replacement:*
> - Don't bloat the list with vague items — but when `## Next` is depleted (< ~6 open items) it is correct to add a fuller batch (6–8) of high-value, well-scoped items so the hourly improver never starves before your next pass. Quality bar is unchanged; only the count floor rises when the queue is thin.

### Proposed edit (improver — empty-backlog fallback)
*Current* (PROMPTS.md step 2, line ~30):
> 2. Pick the top-most unchecked item under "## Next" in STATUS.md.

*Replacement:*
> 2. Pick the top-most unchecked `- [ ]` item under "## Next" in STATUS.md. If there are NO `- [ ]` items under `## Next` (the curator hasn't replenished yet), do not fail or invent a large task: derive ONE small, safe, loop-scoped polish item from the `## Next` subsection reference notes and the current app state, add it as a `- [ ]` under the right subsection, then proceed with it as your pick. Keep it conservative — a clear, single-file improvement — and note in the commit that the backlog was empty and you self-sourced the task.

### Expected effect
The improver stops hitting an empty backlog; the curator keeps ~6–8 items queued
so hourly runs never starve between 6-hourly curator passes.

### Risk / confidence
**Low risk, high confidence.** Both edits preserve safe-scope and the "no
speculative features" bias. The improver fallback is bounded to one small,
self-flagged task. The empty state is observed *right now* in the repo.

## Finding B — `git log … main` returns nothing in cloud clones (daily digest + weekly meta)

### Observations
- Cloud sessions check out a **detached HEAD** at the real tip; the local `main`
  ref is stale. This run: `git rev-parse main` → `340d730`, dated **2026-06-21**,
  **50 commits / 8 days** behind `HEAD`.
- Consequently the literal command `git log --since='7 days ago' … main` (weekly,
  line ~129) and `git log --since='24 hours ago' … main` (digest, line ~99)
  return **empty** — verified by running both in this session.
- This exact hazard is already documented in `NOTES.md` ("Stale local origin/main
  … Always `git fetch origin main` at the start of a run before branching"). The
  digest agent only produces correct output by deviating from its own prompt.

### Diagnosis
Both routine prompts hard-code `main` in a git-log command without first
refreshing the ref, so the literal instruction is wrong in the cloud environment.

### Proposed edit (weekly — line ~129)
*Current:*
> - `git log --since='7 days ago' --date=iso --pretty=format:'%h %ad %s'` on main — commit cadence and types; …

*Replacement:*
> - First `git fetch origin main` (cloud clones check out a detached HEAD with a stale local `main`). Then `git log --since='7 days ago' --date=iso --pretty=format:'%h %ad %s' origin/main` (or `HEAD`) — commit cadence and types; …

### Proposed edit (digest — line ~99)
*Current:*
> - Run `git log --since='24 hours ago' --date=iso --pretty=format:'%h %ad %s'` on main to get commit times + messages.

*Replacement:*
> - First `git fetch origin main` (cloud clones have a stale local `main` ref), then run `git log --since='24 hours ago' --date=iso --pretty=format:'%h %ad %s' origin/main` (or `HEAD`) to get commit times + messages.

### Expected effect
The digest and weekly routines read the true commit history instead of silently
seeing an empty log and risking a "No changes shipped" mis-report.

### Risk / confidence
**Low risk, high confidence.** A read-only command change; matches NOTES.md and
the behavior I had to use in this very run.

## Finding C — Curator's `npx playwright install chromium` is wrong for this sandbox

### Observations
- PROMPTS.md curator Phase 0 step 3 (line ~53) says: "Install a headless browser:
  `npx playwright install chromium` …".
- The execution environment ships Chromium **pre-installed** at
  `PLAYWRIGHT_BROWSERS_PATH=/opt/pw-browsers`, sets `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1`,
  and explicitly states **"Do not run playwright install."** `NOTES.md` independently
  documents the working path (`executablePath` at `/opt/pw-browsers/...`; CDN download blocked).
- The visual pass is *not* failing — the 2026-06-28 digest records **3 visual passes
  (20–40 screenshots each)**. It works only because the curator deviates from its
  prompt per NOTES.md; the literal step is redundant-to-counterproductive.

### Diagnosis
The prompt instructs an install step that the environment forbids and that the
agent must ignore. Aligning it removes a wasted/failing step and a NOTES dependency.

### Proposed edit (curator — Phase 0 step 3, line ~53)
*Current:*
> 3. Install a headless browser: `npx playwright install chromium` (if it fails on system deps, try `npx playwright install --with-deps chromium`).

*Replacement:*
> 3. Use the **pre-installed** Chromium — do NOT run `playwright install` (the CDN download is blocked and the environment sets `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1`). Launch via Playwright's configured `PLAYWRIGHT_BROWSERS_PATH=/opt/pw-browsers`, or pass `executablePath` to the pre-installed Chromium binary under `/opt/pw-browsers/` if auto-detection misses it.

### Expected effect
Removes a forbidden/wasteful install attempt; the visual pass relies on documented
environment state rather than a NOTES.md workaround.

### Risk / confidence
**Low risk, medium-high confidence.** The pass already runs; this just makes the
prompt match the environment. Exact binary path may vary by image — the wording
keeps both the env-var and `executablePath` options.

## Finding D — "Direct to main, no PR" contradicts the observed PR workflow (improver + curator)

### Observations
- Every improver and curator commit this week carries a squash-merge PR suffix:
  `#94`…`#131` (e.g. `d63676b improve: … (#131)`, `d1d50ae curate: … (#127)`).
  That is **38 PR-merged commits** — the loop demonstrably ships via
  `improve/<slug>` branch + PR.
- The daily-digest commit, by contrast, is a **direct push**:
  `fe6932b docs(digest): daily digest 2026-06-28` — no PR number.
- `NOTES.md` states the working rule: *"Branch protection on main: direct
  `git push origin main` returns HTTP 403. All loop commits must go via an
  `improve/<slug>` branch + PR."*
- But PROMPTS.md tells improver/curator the **opposite**: step 10 — "Do NOT create
  a branch and do NOT open a PR"; step 11 — "`git push origin main`"; curator
  Finishing — "commit them directly to main … Do not open a PR and do not create a
  branch."

### Diagnosis
The canonical record (PROMPTS.md) is stale on delivery for the two code-touching
routines: it says direct-to-main while reality (and NOTES.md) is branch+PR. The
loop only succeeds because agents follow NOTES.md over the prompt — fragile, and
it defeats the premise that PROMPTS.md is the source of truth. (The digest's
direct push *succeeds*, so protection appears to gate code/CI paths, not docs —
hence this edit targets improver/curator only, not digest/weekly.)

### Proposed edit (improver — step 10/11, lines ~38–39)
*Current (step 10 tail):*
> … Do NOT create a branch and do NOT open a PR.

*Replacement:*
> … `main` is branch-protected (direct `git push origin main` returns HTTP 403), so deliver via an `improve/<slug>` branch + a PR (GitHub MCP `create_pull_request`) — see NOTES.md "Branch protection on main". Do NOT force-push.

*Current (step 11 head):*
> 11. Sync and push: run `git pull --rebase origin main`, then `git push origin main`. If the push is REJECTED because main advanced under you, …

*Replacement:*
> 11. Sync and deliver: `git fetch origin main`, base your work on the latest `origin/main`, push your `improve/<slug>` branch, and open a PR to main. If the branch falls behind, `git pull --rebase origin main` onto the branch and re-push. If a rebase hits a conflict you cannot cleanly resolve within safe scope, abort (`git rebase --abort`), leave the item under "## In Progress" with a one-line note, and stop — never force-push.

### Proposed edit (curator — Finishing, line ~88)
*Current:*
> … then commit them directly to main with a concise message … Pull/rebase latest main if needed, then push to main. Do not open a PR and do not create a branch.

*Replacement:*
> … then commit them on a `curate/<slug>` branch and open a PR to main (`main` is branch-protected — direct push returns HTTP 403; see NOTES.md). Base the branch on the latest `origin/main` (`git fetch origin main` first) and never force-push.

### Expected effect
PROMPTS.md stops contradicting the enforced workflow; a fresh agent that trusts
the prompt over NOTES.md no longer 403s and aborts.

### Risk / confidence
**Medium confidence — human should verify branch-protection config before applying.**
The direction is unambiguous from 38 PR merges + NOTES.md, but the digest's
successful direct push shows protection is path-dependent; confirm the exact rule
(and whether a loop harness, not the agent, opens the PRs) before editing the live
triggers. This **strengthens** the review gate — it does not weaken any safety
constraint. *Live data point from this run:* the delivery method that actually
lands this proposal commit is recorded in the commit/PR for this file.
